# 장바구니 담기 구현 프로세스 정리

## 현재 결론

현재 장바구니 담기 기능은 `최소 기능 기준 구현 완료` 상태다.

- 구현된 것
  - 장바구니 전역 상태 store 생성
  - 공통 확인 다이얼로그 생성
  - 상품 카드 장바구니 버튼과 다이얼로그 연결
  - 확인 클릭 시 장바구니 데이터 추가
  - 헤더 장바구니 아이콘과 수량 뱃지 연결
  - `/cart` 라우트 연결
  - 장바구니 페이지에서 목록, 수량 변경, 삭제, 총 금액 출력

즉, 지금은 "상품 카드에서 담고, 헤더에서 수량을 확인하고, 장바구니 페이지에서 목록을 관리하는 흐름이 동작하는 상태"라고 보는 게 맞다.

---

## 이번 단계에서 목표로 잡은 최소 기능

가장 먼저 완성해야 하는 흐름은 아래다.

1. 상품 카드에서 장바구니 버튼 클릭
2. `"장바구니에 담을까요?"` 확인 다이얼로그 노출
3. 확인 클릭 시 장바구니 데이터 추가
4. 헤더 장바구니 수량 반영
5. 장바구니 페이지에서 담긴 목록 출력

이 단계까지 되면 주문 기능 전의 핵심 진입 흐름은 살아난다.

---

## 왜 이 순서로 가는가

주문 기능은 바로 붙이기보다 장바구니를 먼저 안정적으로 만들어두는 것이 맞다.

이유:

- 상품 리스트 화면에서 바로 사용자 액션을 만들 수 있음
- 주문서 페이지 전에 상품 선택 흐름을 검증할 수 있음
- 이후 주문 생성 기능을 붙일 때 장바구니 데이터를 그대로 재사용 가능
- 현재 프로젝트 구조에서 가장 작은 단위로 빠르게 확인 가능한 기능임

## 실제 구현 순서

### 1단계. 장바구니 상태 저장 방식 확정

현재는 아래 방식으로 가는 것이 맞다.

- `zustand + persist(localStorage)`

이유:

- 지금 프로젝트 인증 상태 관리와 결이 맞음
- 서버 API 없이도 장바구니 UX를 먼저 완성 가능
- 이후 서버 연동 전환이 비교적 쉬움

### 2단계. 상품 카드 버튼에 다이얼로그 연결

흐름:

- 장바구니 버튼 클릭
- 다이얼로그 열기
- `"확인"` 클릭 시 store에 상품 추가
- 다이얼로그 닫기

### 3단계. 헤더 수량 뱃지 연결

흐름:

- 장바구니 총 수량 구독
- 숫자 자동 반영
- 헤더 아이콘 클릭 시 `/cart` 이동

### 4단계. 장바구니 페이지 라우트 연결

흐름:

- `/cart` 진입
- store에 있는 장바구니 목록 출력
- 수량/삭제 변경 가능

### 5단계. 수동 테스트

체크해야 할 항목:

- 같은 상품 2번 담기 시 수량 증가 여부
- 새로고침 후 장바구니 유지 여부
- 삭제 후 목록 반영 여부
- 수량 감소 시 0 이하 처리 여부
- 장바구니 비었을 때 empty 화면 표시 여부

## 이번 단계에서 중요한 판단

### 확인 다이얼로그를 쓰는 게 맞나

지금 단계에서는 써도 된다.

이유:

- 공통 다이얼로그 컴포넌트를 먼저 확보할 수 있음
- 흐름 검증이 명확함
- 사용자가 장바구니 액션을 인지하기 쉬움

다만 실서비스 UX만 보면 나중에는 아래 형태로 바꿀 가능성이 높다.

- 클릭 즉시 추가
- 토스트 `"장바구니에 담았습니다"`
- `"장바구니로 이동"` 액션 제공

즉, 현재는 구현 검증용으로 다이얼로그 방식이 괜찮고,
추후 UX 다듬기 단계에서 토스트형으로 바꾸는 것을 고려하면 된다.

# 장바구니 관련 테이블 설계 개념적 이해

## 결론부터

장바구니 테이블 설계는 아래 구조로 이해하면 된다.

- `users`
- `carts`
- `products`
- `cart_items`

핵심은 `carts` 와 `products` 를 직접 다대다로 붙이지 않고,
중간에 `cart_items` 테이블을 두어 관계를 푸는 것이다.

이 방식이 장바구니 설계에서 가장 일반적이고 실무적인 방식이다.

---

## 왜 중간 테이블이 필요한가

장바구니는 단순히 "어떤 상품이 담겼다"만 저장하면 끝나지 않는다.

장바구니에 담긴 상품마다 추가로 필요한 정보가 생긴다.

- 몇 개 담았는지 `quantity`
- 언제 담았는지 `created_at`
- 마지막 수정 시점 `updated_at`

나중에는 더 붙을 수도 있다.

- 선택 여부
- 옵션 정보
- 즉시구매 여부
- 행사 가격 기준

즉, 장바구니 안의 상품은 단순 연결 관계가 아니라
"장바구니에 담긴 상품 한 줄"이라는 별도 의미를 가진다.

그래서 `cart_items`를 별도 테이블로 빼는 것이 맞다.

---

## 관계를 어떻게 봐야 하나

겉으로 보면 장바구니와 상품은 다대다 관계다.

- 하나의 장바구니에는 여러 상품이 들어갈 수 있다
- 하나의 상품은 여러 사람 장바구니에 들어갈 수 있다

하지만 DB에서는 이 다대다를 그대로 두지 않고
중간 테이블로 풀어서 1:N + N:1 구조로 표현한다.

즉 실제 관계는 아래처럼 이해하면 된다.

- `carts` 1 : N `cart_items`
- `products` 1 : N `cart_items`

그리고 결과적으로:

- `carts` N : M `products`

이 관계를 `cart_items`가 풀어주는 구조다.

---

## 테이블별 역할

### 1. products

상품 자체를 관리하는 테이블이다.

예시 컬럼:

- `id`
- `name`
- `description`
- `price`
- `stock_quantity`
- `category`
- `image_url`
- `status`

역할:

- 상품 원본 정보 보관
- 장바구니뿐 아니라 주문, 관리자 상품 관리 등에서도 공통 사용

### 2. carts

회원의 장바구니 자체를 나타내는 테이블이다.

예시 컬럼:

- `id`
- `user_id`
- `created_at`
- `updated_at`

역할:

- "누구의 장바구니인가"를 나타냄
- 장바구니 한 개의 상위 컨테이너 역할

현재 프로젝트에서는 보통 아래 정책이 자연스럽다.

- 회원 1명당 장바구니 1개

그렇다면 `carts.user_id`는 사실상 unique로 운영하는 것이 자연스럽다.

### 3. cart_items

장바구니 안에 담긴 각각의 상품 줄을 나타내는 테이블이다.

예시 컬럼:

- `id`
- `cart_id`
- `product_id`
- `quantity`
- `created_at`
- `updated_at`

역할:

- 어떤 장바구니에
- 어떤 상품이
- 몇 개 담겼는지

를 표현하는 핵심 테이블이다.

즉, 장바구니 설계의 중심은 사실 `cart_items`라고 봐도 된다.

---

## 왜 `@ManyToMany` 직접 매핑보다 이 방식이 좋은가

JPA 기준으로도 `Cart` 와 `Product` 를 직접 `@ManyToMany`로 연결하는 방식은
장바구니 같은 도메인에서는 보통 잘 쓰지 않는다.

이유:

- 수량 컬럼을 자연스럽게 넣기 어려움
- 중간 엔티티의 의미가 사라짐
- 나중에 확장할 때 구조가 답답해짐
- 실무에서는 결국 중간 테이블 엔티티를 다시 만들게 됨

그래서 보통은 이렇게 엔티티를 나눈다.

- `Cart`
- `CartItem`
- `Product`

즉, DB 설계도 그렇고 JPA 설계도 그렇고
`cart_items`를 독립 엔티티로 보는 것이 정석에 가깝다.

## 실무적으로 추가하면 좋은 제약과 규칙

### 1. 회원당 장바구니 1개 정책

그렇다면 아래를 고려할 수 있다.

- `carts.user_id` unique

의미:

- 한 회원이 여러 장바구니를 동시에 갖지 않도록 제한

### 2. 같은 상품 중복 row 방지

보통 같은 장바구니에 같은 상품을 여러 줄로 넣지 않는다.
수량만 증가시키는 방식이 일반적이다.

그래서 아래 제약이 유용하다.

- unique(`cart_id`, `product_id`)

의미:

- 같은 장바구니 안에서 같은 상품은 한 줄만 존재

### 3. 인덱스

권장 인덱스:

- `carts.user_id`
- `cart_items.cart_id`
- `cart_items.product_id`

이유:

- 회원 장바구니 조회
- 장바구니 목록 조회
- 상품 기준 참조

에서 자주 사용되기 때문이다.

### 4. 수량 제약

권장:

- `quantity > 0`

의미:

- 0개 이하 데이터가 들어가지 않게 방지

## 현재 프로젝트에 맞춰 해석하면

현재 프로젝트의 흐름에서는 아래 해석이 가장 자연스럽다.

- `users` 1 : 1 `carts`
- `carts` 1 : N `cart_items`
- `products` 1 : N `cart_items`

즉 사용자 한 명이 장바구니 하나를 가지고,
그 장바구니 안에 여러 상품 줄이 들어가는 구조다.

장바구니 페이지에서는 결국 `cart_items`를 기준으로 목록을 그리게 된다.

예를 들면 화면에서 보이는 한 줄은 보통 아래 조합이다.

- `cart_items.quantity`
- `products.name`
- `products.price`
- `products.image_url`

즉, 장바구니 목록은 `cart_items`와 `products`를 조합해서 만드는 화면이라고 이해하면 된다.

## 주문과의 차이

장바구니와 주문은 비슷해 보여도 역할이 다르다.

### 장바구니

- 현재 담아두는 임시 상태
- 상품 최신 정보를 계속 참조해도 됨
- 수량 변경/삭제가 자유로움

### 주문

- 결제가 연결되는 확정 상태
- 당시 가격과 상품명을 snapshot으로 저장해야 함
- 주문 이후에는 상품 정보가 바뀌어도 주문 데이터는 유지되어야 함

즉:

- 장바구니는 `products`를 참조하는 현재 상태
- 주문은 `order_items`에 당시 정보를 복사해 두는 확정 상태

이 차이를 이해하면 왜 장바구니는 단순 참조 구조이고,
주문은 snapshot 구조로 가야 하는지도 자연스럽게 연결된다.

---

## 한 줄 정리

장바구니 설계에서

- `carts`
- `cart_items`
- `products`

구조를 두고,
`cart_items`로 다대다를 풀어내는 방식은
교과서적이면서도 실무적인 정석 설계라고 보면 된다.

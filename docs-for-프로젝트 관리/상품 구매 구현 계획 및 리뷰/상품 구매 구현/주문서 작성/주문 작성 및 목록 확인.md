# 주문 작성 및 목록 확인

## 결론부터

장바구니 다음 단계는 바로 `결제`가 아니라 `주문서 작성`이다.

정확한 흐름은 아래가 맞다.

1. 장바구니에서 주문할 상품 선택
2. 주문하기 버튼 클릭
3. 주문서 작성 페이지 이동
4. 배송지 / 수령인 / 요청사항 입력
5. 주문 생성
6. `orders`, `order_items` 저장
7. 생성된 주문 1건에 대해 바로 결제
8. 결제 완료 후 주문 완료 화면 또는 주문 상세 화면 이동
9. 이후 마이페이지 주문 목록에서 확인

즉, `주문 목록에서 체크하고 결제`보다는
`주문서 작성 후 주문 생성 -> 바로 결제 -> 나중에 목록에서 확인`
이 흐름이 기본형으로 더 맞다.

---

## 왜 이런 순서가 맞는가

결제는 주문이 먼저 만들어진 뒤에 붙는 기능이다.

왜냐하면 결제할 대상이 먼저 확정되어야 하기 때문이다.

결제 전에 고정되어야 하는 것:

- 어떤 상품을 살지
- 수량이 몇 개인지
- 상품 총액이 얼마인지
- 배송비가 얼마인지
- 최종 결제 금액이 얼마인지
- 배송지가 어디인지
- 수령인이 누구인지
- 외부에 보여줄 주문번호가 무엇인지

이 정보가 정리되지 않으면 PG 결제 연동 시 필요한
`orderId`, `amount`, `orderName`을 안정적으로 만들 수 없다.

그래서 항상

- 장바구니는 임시 상태
- 주문은 결제 대상이 되는 확정 상태

로 나눠서 봐야 한다.

## 추천 흐름

### 1. 장바구니에서 주문 대상 선택

장바구니에는 여러 상품이 담길 수 있으므로
주문할 대상만 선택할 수 있게 하는 것이 자연스럽다.

이 단계에서 가능한 기능:

- 전체 선택
- 개별 선택
- 선택 상품만 주문하기

즉 체크박스는 주문 목록보다 장바구니에 있는 쪽이 더 자연스럽다.

### 2. 주문서 페이지 이동

예시 경로:

- `/order`
- `/checkout`

이 페이지에서 보여줄 것:

- 선택한 상품 목록
- 상품별 수량
- 상품 금액 합계
- 배송비
- 총 결제 예상 금액

여기서 사용자는 주문에 필요한 정보를 최종 입력한다.

### 3. 주문자 / 배송 정보 입력

주문서 페이지에서 입력할 대표 항목:

- 주문자 이름
- 주문자 연락처
- 수령인 이름
- 수령인 연락처
- 우편번호
- 기본 주소
- 상세 주소
- 배송 요청사항

이 단계가 있어야 실제 주문이 된다.

### 4. 주문 생성

사용자가 주문서에서 최종 확인을 누르면
백엔드에서 주문을 생성한다.

이때 해야 하는 일:

- 상품 상태 확인
- 재고 확인
- 주문 금액 계산
- 배송비 계산
- 주문번호 생성
- `orders` 저장
- `order_items` 저장

이 시점의 주문 상태는 보통 아래처럼 잡는다.

- `PENDING_PAYMENT`

즉, 주문은 만들어졌지만 아직 결제 전인 상태다.

### 5. 바로 결제 진행

주문이 생성된 뒤에는
방금 만들어진 주문 1건에 대해 바로 결제를 진행한다.

이 단계에서 결제창 호출 또는 PG 연동이 붙는다.

즉,

- 주문 생성
- 결제 승인

은 붙어 있지만 서로 다른 단계다.

### 6. 결제 완료 후 화면 이동

결제가 성공하면 보통 아래 중 하나로 이동한다.

- 주문 완료 페이지
- 주문 상세 페이지

이 페이지에서는

- 주문번호
- 주문 상품
- 결제 금액
- 배송지
- 주문 상태

를 보여주면 된다.

### 7. 주문 목록에서 사후 확인

이후 사용자는 마이페이지나 주문내역 페이지에서
과거 주문을 다시 확인한다.

즉, 주문 목록은 결제 전 단계가 아니라
결제 후 주문을 확인하는 단계라고 보는 게 맞다.

## 비추천 흐름

아래 같은 흐름은 기본 쇼핑몰 사용자 UX로는 다소 어색하다.

1. 주문 버튼 클릭
2. 주문 생성
3. 주문 목록 페이지 이동
4. 목록에서 체크박스로 주문 선택
5. 결제

왜 어색한가:

- 주문을 만들고도 바로 결제하지 않아서 흐름이 끊긴다
- 사용자가 "방금 만든 주문이 뭔지" 다시 찾아야 한다
- 주문 목록은 조회용 성격이 강해서 결제 진입점으로는 약하다
- 미결제 주문이 여러 개 쌓이기 시작하면 관리가 복잡해진다

즉 기본 플로우로는 비추천이다.

---

## 체크박스는 어디에 두는 게 맞나

체크박스는 주문 목록보다 장바구니에 두는 게 더 자연스럽다.

### 장바구니 체크박스의 의미

- 어떤 상품을 이번 주문에 포함할지 고른다

### 주문 목록 체크박스의 의미

- 이미 만들어진 주문 여러 개를 다시 골라 처리한다

후자는 일반 사용자 주문 UX보다는
관리자 일괄 처리나 특수 케이스에 가깝다.

그래서 현재 프로젝트에서 체크박스가 필요하다면
우선 장바구니 화면에 두는 것이 맞다.

## 현재 프로젝트에 맞는 구현 순서

### 1단계. 장바구니 체크 기능 추가

장바구니에서 주문할 상품을 고를 수 있게 한다.

예시:

- 상품별 체크박스
- 전체 선택
- 선택 상품 총액 계산

### 2단계. 주문서 페이지 만들기

장바구니에서 선택한 항목만 넘겨서 주문서 페이지를 구성한다.

표시할 내용:

- 선택 상품 목록
- 수량
- 주문 금액
- 배송비
- 최종 합계

### 3단계. 주문자/배송지 입력 폼 만들기

주문에 필요한 배송 정보를 입력하게 한다.

### 4단계. 백엔드 주문 생성 API 만들기

필요 엔드포인트 예시:

- `POST /api/orders`
- `GET /api/orders`
- `GET /api/orders/{orderId}`

### 5단계. 주문 생성 후 바로 결제 연결

이때 결제는 주문 1건 기준으로 연결한다.

### 6단계. 주문 목록 페이지 만들기

이 페이지는 결제 전 처리용이 아니라
결제 후 확인용으로 보는 게 맞다.

표시할 것:

- 주문번호
- 주문일시
- 대표 상품명
- 결제 금액
- 주문 상태

## DB 관점에서 이해

### cart_items

- 아직 담아둔 임시 선택 상태

### orders

- 결제 대상이 되는 확정 주문

### order_items

- 주문 당시 상품 정보 snapshot

### payments

- 실제 결제 이력

즉 흐름은 아래처럼 이어진다.

- `cart_items`에서 선택
- `orders`, `order_items` 생성
- `payments` 생성

이 순서가 가장 자연스럽다.

## 최종 정리

질문:

- `주문 버튼 => 주문서 작성 => 주문서 테이블에 저장 => 주문 페이지로 연결 => 주문 목록 확인 => 체크 박스로 체크 => 결제`

답:

- 방향은 일부 맞지만, 기본 흐름으로는 순서를 수정하는 게 좋다

더 적절한 흐름은 아래다.

1. 장바구니에서 체크
2. 주문서 작성
3. 주문 생성
4. 바로 결제
5. 주문 완료/주문 상세 이동
6. 나중에 주문 목록에서 확인

즉 체크박스는 주문 목록보다 장바구니 쪽이 더 적절하고,
주문 목록은 결제 전 처리 화면이 아니라 결제 후 조회 화면으로 보는 것이 맞다.

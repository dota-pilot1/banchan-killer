# 주문서 작성 2차 구현 계획

## 2차 목표

2차 구현의 목표는
`주문서 UI -> 프로필/배송지 데이터 -> 실제 주문 생성`
까지 이어 붙이는 것이다.

즉 주문서를 단순 화면이 아니라
실제 사용자 정보와 주문 데이터가 연결된 단계로 만드는 것이 핵심이다.

---

## 현재 2차에서 보는 구현 범위

### 포함

- 프로필 페이지 API 기반 전환
- 배송지 API 연동
- 여러 배송지 저장 구조 반영
- 기본 배송지 지정 기능
- 주문서 진입 시 기본 배송지 자동 채움
- 주문서에서 배송지 선택 UI
- 주문 생성 API 연결
- 주문 목록 / 주문 상세 화면 초안

### 제외

- PG 결제창 호출
- 결제 승인 검증
- 결제 실패 / 재시도 정교화
- 재고 차감 확정
- 취소 / 환불

---

## 2차 구현 이유

1차까지만 있으면 주문서는 보기 좋은 화면에 가깝다.
하지만 2차가 되어야 아래가 가능해진다.

- 프로필에 저장한 정보가 주문서에 실제 반영됨
- 기본 배송지를 주문서에서 자동으로 선택 가능
- 배송지2, 배송지3 같은 주소 선택 흐름 가능
- 주문서 입력 내용이 실제 주문 데이터로 저장됨

즉 사용자가 체감하는 "진짜 주문 흐름"은 2차부터 시작된다.

---

## 2차 사용자 흐름

1. 장바구니에서 상품 선택
2. 주문서 페이지 진입
3. 기본 배송지가 먼저 채워짐
4. 필요하면 배송지2 / 배송지3로 변경
5. 주문자 / 수령인 / 요청사항 수정
6. 주문 생성 API 호출
7. 주문 상세 또는 주문 목록으로 이동

---

## 프론트 2차 작업 항목

### 프로필 / 배송지

- `localStorage` 기반이 아니라 API 기반으로 전환
- 마이페이지에서 배송지 여러 개 저장 가능
- 배송지별 기본 배송지 지정 가능
- 주문서에서 저장된 주소 목록 선택 가능

### 주문서

- 기본 배송지 자동 채움
- 선택한 배송지로 폼 즉시 변경
- 주문자 정보와 수령인 정보 수정 가능
- 주문 생성 API 호출
- 성공 시 주문 상세 페이지 이동

### 주문 조회

- 주문 목록 진입점 마련
- 주문 상세 화면 표시
- 주문번호 / 상태 / 주소 / 금액 / 상품 표시

---

## 백엔드 2차 작업 항목

### 프로필 / 배송지

- `GET /api/users/me`
- `PATCH /api/users/me`
- `GET /api/users/me/addresses`
- `POST /api/users/me/addresses`
- `PATCH /api/users/me/addresses/{addressId}`
- `PATCH /api/users/me/addresses/{addressId}/default`
- `DELETE /api/users/me/addresses/{addressId}`

### 주문

- `POST /api/orders`
- `GET /api/orders`
- `GET /api/orders/{orderId}`
- `orders`, `order_items` 엔티티 추가
- 주문 상태 enum 추가
- 주문번호 생성

---

## 2차 완료 기준

아래가 되면 2차 구현 완료로 본다.

1. 프로필과 배송지 정보가 API 기반으로 저장된다.
2. 기본 배송지를 지정할 수 있다.
3. 주문서에서 저장된 배송지 목록을 선택할 수 있다.
4. 선택한 상품 기준으로 실제 주문 생성이 가능하다.
5. 주문 상세 화면 또는 주문 목록에서 생성 결과를 확인할 수 있다.

즉 2차는
`주문서가 실제 사용자 데이터와 주문 데이터에 연결된 상태`
를 의미한다.

---

## 2차 다음 단계

3차에서는 주문을 실제 결제로 넘기는 단계가 필요하다.

다음 우선순위:

1. 결제사 연동
2. 결제 승인 검증
3. `payments` 저장
4. 주문 상태 `PAID` 변경
5. 재고 차감 / 실패 시 롤백 정리

---

`프로필/배송지/주문 생성 API를 붙여서 주문서를 실제 주문 데이터로 만드는 단계`
다.

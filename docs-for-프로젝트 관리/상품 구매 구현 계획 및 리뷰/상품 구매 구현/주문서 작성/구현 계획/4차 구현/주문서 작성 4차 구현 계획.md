# 주문서 작성 4차 구현 계획

## 4차 목표

토스페이먼츠 심사 통과 후 실결제를 활성화하고,
재고 차감/복구 로직을 구현하는 단계.

---

## 4차 범위

### 포함

- 실결제 키 전환 (테스트 → 라이브)
- 결제위젯 방식 전환 (심사 후 위젯 키 발급)
- 재고 차감 (결제 성공 시)
- 재고 복구 (결제 실패/취소 시)
- 동시 주문 시 재고 정합성 (비관적 락 or 낙관적 락)
- 실결제 테스트

### 제외

- 환불/취소 정교화 (5차)
- 배송 상태 관리 (5차)
- 쿠폰/포인트 (6차)

---

## 사용자 흐름 변경 없음

4차는 내부 로직 변경만 있고 사용자 흐름은 3차와 동일.
장바구니 → 주문서 → 결제 → 결제완료 → 주문상세

---

## 백엔드 작업 항목

### 실결제 전환

1. `.env`에서 테스트 키 → 라이브 키 교체
2. 프론트 `.env`에서 `VITE_TOSS_CLIENT_KEY` 교체
3. 코드 변경 없음

### 결제위젯 전환 (선택)

심사 통과 시 위젯 키(`live_gck_`) 발급됨:
1. `PaymentPage.tsx`에서 `tossPayments.widgets()` 방식으로 전환
2. 결제수단 선택 UI가 토스 위젯으로 대체
3. 더 깔끔한 UX 제공

### 재고 차감

```
결제 성공 시:
  orderItems의 각 상품 → product.stockQuantity -= quantity
  stockQuantity < 0 이면 결제 거부

결제 실패/취소 시:
  product.stockQuantity += quantity (복구)
```

구현 위치: `PaymentService.confirmPayment()` 내부

### 동시성 제어

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<Product> findByIdForUpdate(Long id);
```

또는 `@Version` 필드 추가 (낙관적 락)

---

## 프론트 작업 항목

- 결제위젯 방식 전환 (위젯 키 발급 후)
- 재고 부족 시 에러 메시지 표시
- 품절 상품 장바구니에서 표시

---

## 4차 완료 기준

1. 실제 카드로 결제가 완료된다
2. 결제 성공 시 재고가 차감된다
3. 동시 주문 시 재고 초과 판매가 발생하지 않는다
4. 토스 대시보드에서 실거래 내역이 확인된다

---

## 한 줄 정리

4차는 `테스트 결제를 실결제로 전환하고, 재고 정합성을 보장하는 단계`다.

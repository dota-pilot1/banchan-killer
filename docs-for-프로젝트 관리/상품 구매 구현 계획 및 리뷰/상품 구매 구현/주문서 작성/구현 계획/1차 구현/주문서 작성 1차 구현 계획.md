# 주문서 작성 1차 구현 계획

## 1차 목표

1차 구현의 목표는
`장바구니 -> 주문서 페이지 진입 -> 주문 대상과 금액 확인`
까지의 흐름을 먼저 만드는 것이다.

즉 결제나 주문 저장보다 먼저
장바구니 다음에 사용자가 어디서 무엇을 확인해야 하는지
주문 UX의 뼈대를 만드는 단계다.

---

## 1차 범위

### 포함

- 장바구니 상품 체크박스
- 전체 선택 / 개별 선택
- 선택 상품 수 계산
- 선택 상품 총액 계산
- 주문서 페이지 라우트 추가
- 선택 상품 기준 주문서 화면 렌더링
- 주문 상품 목록 요약
- 배송비 / 총액 요약 UI

### 제외

- 프로필 API 연동
- 배송지 API 연동
- 다중 배송지 선택
- 실제 주문 생성 API 호출
- 주문 목록 / 주문 상세
- 결제 연동

---

## 1차 구현 이유

이 단계가 먼저 있어야
장바구니와 주문서가 분리된다.

즉:

- 장바구니는 담아두는 곳
- 주문서는 확정 직전 확인 화면

이라는 역할 구분을 먼저 만들 필요가 있다.

---

## 1차 사용자 흐름

1. 사용자가 장바구니에 상품을 담는다.
2. 장바구니에서 주문할 상품을 체크한다.
3. 선택 상품 기준으로 주문서 페이지로 이동한다.
4. 주문서에서 상품 목록과 금액을 확인한다.

이 단계에서는 아직 저장된 배송지나 실제 주문 생성은 없어도 된다.

---

## 프론트 1차 작업 항목

### 장바구니

- `selectedProductIds` 상태 추가
- 개별 선택 / 전체 선택 처리
- 선택된 상품 수 표시
- 선택 상품 금액 계산
- `주문하기` 버튼 연결

### 주문서 페이지

- `/order` 라우트 추가
- 선택 상품만 표시
- 상품 이미지 / 이름 / 수량 / 금액 표시
- 상품 금액 / 배송비 / 최종 금액 표시

---

## 1차 완료 기준

아래가 되면 1차 구현 완료로 본다.

1. 장바구니에서 상품 선택 가능
2. 선택 상품만 기준으로 주문서 페이지 이동 가능
3. 주문 상품 목록 표시 가능
4. 금액 요약 표시 가능

즉 1차는
`주문서 진입 UX의 뼈대 완성`
정도로 보면 된다.

---

## 1차 다음 단계

2차에서는 주문서가 실제 사용자 정보와 연결되어야 한다.

다음 우선순위:

1. 프로필 / 배송지 API 연동
2. 기본 배송지 자동 채움
3. 다중 배송지 선택
4. 주문 생성 API 연결

---

## 한 줄 정리

1차 구현은
`장바구니에서 주문 대상을 고르고 주문서 화면에서 확인할 수 있게 만드는 단계`
다.

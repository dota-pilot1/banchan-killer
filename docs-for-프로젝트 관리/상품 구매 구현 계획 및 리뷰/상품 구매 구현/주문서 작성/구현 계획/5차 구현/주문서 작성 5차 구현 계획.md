# 주문서 작성 5차 구현 계획

## 5차 목표

주문 취소/환불, 배송 상태 관리, 관리자 주문 처리 등
서비스 운영에 필요한 핵심 기능을 구현하는 단계.

---

## 5차 범위

### 포함

- 주문 취소 (사용자)
- 결제 환불 (토스 취소 API 연동)
- 배송 상태 관리 (PREPARING → SHIPPED → DELIVERED)
- 관리자 주문 처리 대시보드
- 주문 상태 전이 규칙 정립

### 제외

- 부분 취소/부분 환불 (6차)
- 쿠폰/포인트 (6차)
- 비회원 주문 (6차)

---

## 주문 상태 전이 (State Machine)

```
PENDING_PAYMENT → PAID → PREPARING → SHIPPED → DELIVERED
                    ↓
                 CANCELLED (취소)
                    ↓
                 REFUNDED (환불 완료)
```

규칙:
- PENDING_PAYMENT: 결제 전 취소 가능 (무조건)
- PAID: 배송 준비 전 취소 가능 → 토스 환불 API 호출
- PREPARING: 관리자가 배송 준비 중 표시
- SHIPPED: 배송 시작 → 취소 불가 (반품으로 전환)
- DELIVERED: 배송 완료

## 백엔드 작업 항목

### 주문 취소

```
POST /api/orders/{orderId}/cancel
```

처리:
1. 주문 상태 확인 (PAID/PENDING_PAYMENT만 취소 가능)
2. PAID 상태면 토스 환불 API 호출
3. 재고 복구
4. 주문 상태 → CANCELLED
5. payments 상태 → CANCELLED

### 토스 환불 API

```
POST https://api.tosspayments.com/v1/payments/{paymentKey}/cancel
Body: { "cancelReason": "고객 요청" }
```

### 배송 상태 관리

```
PATCH /api/admin/orders/{orderId}/status
Body: { "status": "SHIPPED", "trackingNumber": "1234567890" }
```

관리자 전용 API.

### OrderStatus 확장

```java
public enum OrderStatus {
    PENDING_PAYMENT,
    PAID,
    PREPARING,
    SHIPPED,
    DELIVERED,
    CANCELLED,
    REFUNDED
}
```

## 프론트 작업 항목

### 사용자

- 주문 상세에서 "주문 취소" 버튼 (PAID/PENDING_PAYMENT 상태만)
- 취소 사유 입력 모달
- 주문 목록에서 상태 뱃지 (배송중, 배송완료, 취소됨 등)
- 배송 추적 번호 표시

### 관리자

- 관리자 주문 목록 페이지 (`/admin/orders`)
- 주문 상태 변경 드롭다운 (PREPARING → SHIPPED → DELIVERED)
- 송장번호 입력
- 취소/환불 처리

---

## 5차 완료 기준

1. 사용자가 주문을 취소할 수 있다
2. PAID 상태 취소 시 토스 환불이 자동 처리된다
3. 관리자가 배송 상태를 변경할 수 있다
4. 주문 목록에서 상태별 필터링이 가능하다
5. 배송 추적 번호가 표시된다

---

## 한 줄 정리

5차는 `주문 취소/환불 + 배송 관리 + 관리자 기능으로 서비스 운영이 가능한 상태를 만드는 단계`다.

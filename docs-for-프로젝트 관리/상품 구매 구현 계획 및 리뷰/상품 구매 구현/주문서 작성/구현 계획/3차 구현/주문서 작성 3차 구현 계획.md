# 주문서 작성 3차 구현 계획

## 3차 목표

3차 구현의 목표는
`생성된 주문 -> 실제 결제 -> 주문 상태 확정`
까지 마무리하는 것이다.

즉 2차에서 만든 주문 데이터를
실제 결제 가능한 주문으로 완성하는 단계다.

---

## 3차 범위

### 포함

- 결제사(PG) 연동
- 결제 요청 파라미터 구성
- 결제 승인 결과 검증
- `payments` 저장
- 주문 상태 `PENDING_PAYMENT -> PAID` 변경
- 결제 성공 / 실패 / 취소 화면 정리
- 재고 차감 시점 확정

### 제외

- 부분 취소 / 부분 환불
- 포인트 / 쿠폰 / 할인 정책
- 관리자 정산 / 매출 통계
- 비회원 주문

---

## 3차 구현 이유

2차까지는 주문이 생성되어도
아직 돈이 오간 주문은 아니다.

3차가 필요한 이유는 아래와 같다.

- 실제 결제 금액을 검증해야 함
- 결제 성공 여부에 따라 주문 상태를 바꿔야 함
- 주문과 결제 데이터를 분리 저장해야 함
- 이후 취소/환불 기능의 기준 데이터를 만들어야 함

즉 3차부터 주문 플로우가 서비스 운영 수준에 가까워진다.

---

## 3차 사용자 흐름

1. 사용자가 주문서를 작성한다.
2. 주문 생성 API 호출로 주문번호를 발급받는다.
3. 결제창 또는 결제 단계로 이동한다.
4. 결제 성공 시 주문 상태를 `PAID`로 변경한다.
5. 결제 실패 시 주문 상태는 `PENDING_PAYMENT` 또는 실패 상태로 유지한다.
6. 사용자는 주문 목록 / 주문 상세에서 최종 결과를 확인한다.

---

## 프론트 3차 작업 항목

### 결제 진입

- 주문 생성 성공 후 결제 파라미터 준비
- 결제창 호출 버튼 연결
- 결제 성공 / 실패 / 취소 결과 페이지 처리

### 주문 상태 반영

- 결제 성공 시 주문 상세 갱신
- 주문 목록 상태 뱃지 반영
- 실패 시 재결제 또는 재시도 동선 정리

---

## 백엔드 3차 작업 항목

### 결제

- 결제 승인 API 연동
- 결제 요청 위변조 검증
- `payments` 엔티티 / 테이블 추가
- 결제 키 / 거래번호 / 승인 시각 저장

### 주문 상태 처리

- 결제 성공 시 `PAID`
- 결제 실패 시 상태 유지 또는 실패 상태 기록
- 중복 승인 방지
- 실패 시 재고 / 주문 정합성 검토

---

## 3차 완료 기준

아래가 되면 3차 구현 완료로 본다.

1. 주문 생성 후 실제 결제 단계로 이동 가능
2. 결제 승인 결과를 백엔드에서 검증 가능
3. 결제 정보 저장 가능
4. 주문 상태가 결제 결과에 따라 변경됨
5. 사용자 화면에서 결제 완료 여부를 확인 가능

즉 3차는
`주문이 실제 결제된 주문으로 확정되는 단계`
다.

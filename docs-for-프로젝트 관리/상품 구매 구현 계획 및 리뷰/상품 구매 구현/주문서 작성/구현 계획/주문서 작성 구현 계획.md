# 주문서 작성 구현 계획

## 목표

장바구니 다음 단계에서 사용자가 바로 결제 화면으로 뛰지 않고,
주문에 필요한 정보를 먼저 정리하는 `주문서 작성 단계`를 만든다.

이 단계의 목표는 아래다.

- 장바구니에서 주문할 상품 선택
- 주문서 페이지 이동
- 주문자 / 수령인 / 배송지 정보 입력
- 최종 주문 금액 계산
- 주문 생성 API 호출
- 생성된 주문 1건을 결제 단계로 넘기기

즉, 주문서 작성 기능은 결제 전에 주문을 확정하는 단계다.

---

## 권장 사용자 흐름

### 1. 장바구니에서 주문 상품 선택

장바구니에 담긴 모든 상품을 항상 주문하지 않을 수도 있으므로
체크박스로 주문 대상을 고를 수 있게 하는 것이 좋다.

예시:

- 전체 선택
- 개별 선택
- 선택한 상품만 주문하기

### 2. 주문하기 버튼 클릭

장바구니에서 선택한 상품 기준으로 주문서 페이지로 이동한다.

예상 경로:

- `/order`
- 또는 `/checkout`

### 3. 주문서 페이지에서 정보 입력

입력 항목 예시:

- 주문자 이름
- 주문자 연락처
- 수령인 이름
- 수령인 연락처
- 우편번호
- 기본 주소
- 상세 주소
- 배송 요청사항

### 4. 주문 금액 확인

주문서 페이지에서 아래 내용을 한 번에 보여준다.

- 주문 상품 목록
- 수량
- 상품 금액 합계
- 배송비
- 최종 결제 금액

### 5. 주문 생성

사용자가 주문 내용을 확인하고 최종 버튼을 누르면
백엔드에 주문 생성 요청을 보낸다.

### 6. 결제 단계 이동

주문 생성이 완료되면
해당 주문 1건을 기준으로 결제 단계로 이동한다.

즉:

- 주문서 작성
- 주문 생성
- 결제

이 순서가 된다.

---

## 구현 범위

### 프론트

- 장바구니 페이지에 체크박스 추가
- 선택 상품만 주문서로 넘기기
- 주문서 페이지 UI 구현
- 주문자/배송지 입력 폼 구현
- 금액 요약 영역 구현
- 주문 생성 API 연결

### 백엔드

- 주문 생성 요청 DTO 정의
- 주문 생성 API 구현
- 주문 조회 API 구현
- `orders`, `order_items` 엔티티 추가
- 주문 상태 enum 정의

---

## 프론트 구현 단계

### 1단계. 장바구니 선택 기능 추가

현재 장바구니는 목록과 수량 변경까지만 있다.
여기에 주문 대상을 선택하는 UI를 추가해야 한다.

필요 요소:

- 상품별 체크박스
- 전체 선택
- 선택된 상품 수 계산
- 선택 상품 총액 계산

### 2단계. 주문서 페이지 라우트 추가

예시:

- `/order`

주문서 페이지는 장바구니의 선택된 항목만 기준으로 렌더링해야 한다.

### 3단계. 주문서 입력 폼 구현

주문서 페이지에는 아래 영역이 필요하다.

- 주문 상품 요약
- 주문자 정보 입력
- 배송 정보 입력
- 요청사항 입력
- 금액 요약

### 4단계. 주문 생성 요청 연결

주문 생성 버튼 클릭 시 프론트가 백엔드에 아래 정보를 보낸다.

- 선택한 상품 목록과 수량
- 주문자 정보
- 배송지 정보
- 요청사항

### 5단계. 주문 생성 성공 후 화면 이동

주문 생성 성공 시:

- 주문 완료 페이지로 이동
- 또는 결제 시작 화면으로 이동

현재 프로젝트 흐름에서는
주문 생성 후 바로 결제 단계로 넘어가는 쪽이 더 자연스럽다.

---

## 백엔드 구현 단계

### 1단계. 주문 상태 정의

초기 상태 예시:

- `PENDING_PAYMENT`
- `PAID`
- `CANCELLED`

이후 확장:

- `PREPARING`
- `SHIPPED`
- `DELIVERED`
- `REFUNDED`

### 2단계. orders 엔티티 추가

주문 상위 정보 저장:

- 주문번호
- 회원 ID
- 주문 상태
- 주문자 정보
- 수령인 정보
- 주소 정보
- 금액 정보

### 3단계. order_items 엔티티 추가

주문 당시 상품 정보를 snapshot으로 저장한다.

이유:

- 상품 가격이 나중에 바뀌어도 과거 주문 데이터는 유지해야 함
- 상품명이 바뀌어도 주문 당시 정보는 남아야 함

### 4단계. 주문 생성 서비스 구현

주문 생성 시 해야 할 일:

- 선택 상품 재고 확인
- 상품 상태 확인
- 금액 계산
- 배송비 계산
- 주문번호 생성
- `orders`, `order_items` 저장

### 5단계. 주문 조회 API 구현

필요 API 예시:

- `POST /api/orders`
- `GET /api/orders`
- `GET /api/orders/{orderId}`

## 화면 설계 포인트

### 주문서 페이지에서 꼭 보여줘야 하는 것

- 상품 썸네일
- 상품명
- 수량
- 주문 금액
- 배송비
- 최종 합계

### 사용자가 바로 이해해야 하는 것

- 지금 어떤 상품을 주문하는지
- 배송 정보를 어디에 입력하는지
- 최종 얼마를 결제하는지
- 다음 단계가 결제라는 점

즉, 주문서 페이지는 "입력 폼"이라기보다
"주문 확정 전 최종 확인 화면"처럼 보이게 만드는 게 좋다.

---

## 체크박스는 어디에 둘 것인가

기본 흐름에서는 주문 목록보다 장바구니에 두는 것이 맞다.

이유:

- 주문 전 선택 단계는 장바구니에서 끝내는 것이 자연스럽다
- 주문 목록은 이미 생성된 주문을 확인하는 페이지 성격이 강하다

즉 추천 구조는 아래다.

- 장바구니: 체크박스로 주문 대상 선택
- 주문서: 선택된 항목 최종 확인
- 주문 목록: 결제 후 내역 조회

---

## 주문 목록 페이지의 역할

주문 목록은 결제 전 처리 화면이 아니라
결제 후 조회 화면으로 보는 것이 맞다.

표시 항목 예시:

- 주문번호
- 주문일시
- 대표 상품명
- 총 금액
- 주문 상태

주문 목록에서 할 수 있는 것:

- 주문 상세 보기
- 배송 상태 확인
- 취소 가능 상태면 취소 요청

하지만 기본형에서는
주문 목록에서 체크하고 결제하는 구조는 비추천이다.

## 최종 정리

주문서 작성 기능은
장바구니와 결제 사이를 연결하는 핵심 단계다.

흐름은 아래처럼 가는 것이 가장 자연스럽다.

1. 장바구니 선택
2. 주문서 작성
3. 주문 생성
4. 바로 결제
5. 주문 완료 / 주문 목록 확인

즉 다음 단계는
`주문서 작성 기능 구현`
이 맞다.

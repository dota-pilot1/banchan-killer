# 주문서 작성 구현 계획

## 문서 재정리 목적

주문서 작성 기능은 처음에는 한 번에 보였지만,
실제로 구현해보면 아래처럼 단계가 분리된다.

1. 장바구니에서 주문 대상을 고르는 단계
2. 주문서 UI와 프로필/배송지 연동 단계
3. 주문 생성과 조회 API를 붙이는 단계
4. 실제 결제 연동 단계
5. 취소/환불/운영 고도화 단계

그래서 지금 문서는 `1차`, `2차`, `3차` 기준으로 나누고,
필요하면 이후 `4차`, `5차`로 확장하는 기준 문서로 본다.

---

## 현재 진행 기준

현재까지 구현 상태는 대략 아래처럼 보는 것이 맞다.

### 1차 구현 완료 범위

- 장바구니 선택 체크박스 추가
- 선택 상품만 주문 흐름으로 넘기기
- 주문서 페이지 진입
- 주문 상품 요약 표시
- 주문 금액 요약 표시

### 2차 구현 진행 / 대부분 연결 범위

- 프로필 페이지 API 기반 전환
- 배송지 API 연동
- 기본 배송지 자동 채움
- 여러 배송지 저장 및 기본 배송지 지정 구조 반영
- 주문서에서 배송지 선택 UI 반영
- 주문 생성 API 연결
- 주문 목록 / 주문 상세 라우트 및 화면 초안 반영

즉 지금 시점은
`주문서 UI만 있는 상태`는 이미 넘었고,
`프로필/배송지/주문 생성이 연결되는 2차 단계`로 보는 것이 맞다.

---

## 단계별 큰 구분

### 1차 구현

주문서를 쓸 수 있게 만드는 단계

- 장바구니 선택
- 주문서 페이지 진입
- 주문 상품/금액 확인
- 주문 입력 폼 초안

### 2차 구현

주문서가 실제 사용자 데이터와 연결되는 단계

- 프로필 API 연동
- 배송지 API 연동
- 기본 배송지 자동 채움
- 다중 배송지 선택
- 주문 생성 API 연동
- 주문 목록 / 주문 상세 진입점 구성

### 3차 구현

주문을 실제 결제로 넘기고 운영 가능 상태로 만드는 단계

- PG 연동
- 결제 승인 검증
- 결제 결과 저장
- 주문 상태 변경
- 재고 차감/복구 정책 정리

---

## 왜 3차까지 나누는가

이렇게 나누면 구현 우선순위가 명확해진다.

- 1차: 장바구니 다음 UX 만들기
- 2차: 사용자 정보와 주문 데이터를 실제로 연결하기
- 3차: 결제/운영 로직으로 확정하기

특히 지금 프로젝트처럼 혼자 빠르게 구현할 때는
문서를 너무 세세한 기술 목록으로 적는 것보다
`어느 단계까지 끝났는지`를 바로 판단할 수 있게 나누는 편이 훨씬 유용하다.

---

## 이후 확장 기준

필요하면 아래처럼 더 나눠도 된다.

### 4차 구현 예시

- 주문 취소
- 환불 처리
- 배송 상태 조회
- 관리자 주문 관리

### 5차 구현 예시

- 쿠폰 / 포인트
- 주소 검색 API 연동
- 다중 결제수단
- 비회원 주문
- 정기배송/재주문

즉 지금은 3차까지 나누고,
운영성 기능이 커질 때 4차, 5차를 추가하는 방식이 가장 관리하기 좋다.

---

## 한 줄 정리

지금까지 구현은 `2차 단계까지 진행 중`으로 정리하는 것이 맞고,
문서는 `1차, 2차, 3차` 기준으로 나눠두는 편이 이후 구현과 회고에 가장 도움이 된다.

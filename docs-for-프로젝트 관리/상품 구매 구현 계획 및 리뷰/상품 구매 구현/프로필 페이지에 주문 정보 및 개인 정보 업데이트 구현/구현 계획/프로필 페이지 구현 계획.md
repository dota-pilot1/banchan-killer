# 프로필 페이지 구현 계획

## 목표

프로필 페이지는 단순 회원 정보 조회 화면이 아니라,
사용자가 주문 전에 필요한 기본 정보를 미리 관리하는 화면으로 만든다.

핵심 목표는 아래다.

- 헤더 사용자 영역에서 프로필 진입 가능
- 마이페이지에서 개인 정보 확인 및 수정 가능
- 배송 정보를 미리 저장 가능
- 주문서 작성 시 저장된 배송 정보를 자동 반영 가능
- 주문 내역을 이후에 프로필 영역에서 확인 가능

즉, 주문서에서 매번 정보를 처음부터 입력하게 하기보다
프로필 페이지에서 미리 관리하고 주문 단계에서는 불러와 확인하는 구조로 간다.

---

## 왜 프로필 페이지가 먼저 필요한가

주문 단계에서 아래 정보를 매번 직접 쓰게 하면 UX가 불편하다.

- 이름
- 연락처
- 배송지
- 배송 요청사항 기본값

이 정보는 주문마다 완전히 달라지기보다
대부분 비슷한 값이 반복된다.

그래서 교과서적인 방향은 아래에 가깝다.

- 프로필/마이페이지에서 기본 정보 관리
- 배송지 관리에서 기본 배송지 저장
- 주문서에서는 자동 채움
- 필요할 때만 수정

이 구조가 더 직관적이고 재주문 UX도 좋아진다.

---

## 권장 UX 구조

### 1. 헤더 사용자 영역 드롭다운

현재 상단 우측에 사용자명 영역이 있으므로
이 부분을 클릭하면 드롭다운이 열리게 한다.

추천 메뉴:

- `프로필`
- `주문 내역`
- `배송지 관리`
- `로그아웃`

초기에는 `프로필`과 `로그아웃`만 먼저 열어도 된다.

### 2. 프로필 페이지 진입

예상 경로:

- `/mypage`
- 또는 `/profile`

현재 프로젝트 성격상 `mypage`가 더 자연스럽다.

### 3. 좌측 탭 / 우측 상세 구조

추천 레이아웃:

- 왼쪽: 탭 메뉴
- 오른쪽: 현재 선택된 상세 정보 영역

초기 탭 구성 추천:

1. `기본 정보`
2. `배송지 관리`

추후 확장 탭:

- `주문 내역`
- `결제 수단`
- `알림 설정`

즉, 지금은 2탭 구조로 시작하는 것이 적절하다.

---

## 탭별 기능 정의

### 1. 기본 정보 탭

보여줄 정보:

- 프로필 이미지
- 닉네임
- 이메일
- 연락처

수정 가능한 정보:

- 프로필 이미지
- 닉네임
- 연락처

수정 불가 또는 제한적으로 다룰 정보:

- 이메일
- 권한(role)

초기 버전에서는 비밀번호 변경은 분리해도 된다.

### 2. 배송지 관리 탭

핵심 목적:

- 주문서에서 자동 반영할 기본 배송지 관리

보여줄 정보:

- 배송지 목록
- 기본 배송지 여부
- 수령인 이름
- 연락처
- 주소
- 배송 요청사항 기본값

가능한 동작:

- 배송지 추가
- 배송지 수정
- 배송지 삭제
- 기본 배송지 지정

현재 프로젝트에서는 최소 범위로 아래만 먼저 해도 충분하다.

- 배송지 1개 저장
- 기본 배송지 1개 지정

즉 처음부터 주소록을 복잡하게 만들기보다
기본 배송지 관리부터 시작하는 것이 좋다.

### 3. 주문 내역 탭 또는 별도 페이지

이 기능은 초기 2탭 안에 넣어도 되고,
일단은 별도 페이지로 두었다가 나중에 통합해도 된다.

주문 내역에서 보여줄 것:

- 주문번호
- 주문일시
- 대표 상품명
- 주문 금액
- 주문 상태

현재 우선순위로 보면

- 기본 정보
- 배송지 관리

가 먼저고,
주문 내역은 그 다음이다.

---

## 주문서와의 연결 방식

프로필 페이지를 만드는 이유는
주문서 작성 단계를 더 가볍게 만들기 위해서다.

권장 흐름:

1. 사용자가 프로필 페이지에서 기본 배송지 저장
2. 장바구니에서 주문하기 클릭
3. 주문서 페이지 진입
4. 저장된 기본 배송지 자동 채움
5. 사용자는 필요한 부분만 수정
6. 주문 생성 후 결제 단계 이동

즉 주문서는 "완전한 신규 입력 페이지"가 아니라
"저장된 정보를 불러와 확인/수정하는 페이지"가 되는 것이 맞다.

## 프론트 구현 단계

### 1단계. 헤더 사용자 드롭다운

필요 기능:

- 사용자명 클릭
- 드롭다운 열기/닫기
- 프로필 진입
- 로그아웃

### 2단계. 마이페이지 라우트 추가

예시:

- `/mypage`

### 3단계. 2컬럼 레이아웃 구현

구조:

- 왼쪽 탭 메뉴
- 오른쪽 상세 패널

### 4단계. 기본 정보 탭 구현

구성:

- 프로필 이미지 카드
- 닉네임 입력
- 이메일 표시
- 연락처 입력
- 저장 버튼

### 5단계. 배송지 관리 탭 구현

구성:

- 배송지 목록 또는 기본 배송지 카드
- 수령인 이름
- 연락처
- 주소 입력
- 상세 주소 입력
- 기본 배송지 설정
- 저장 버튼

### 6단계. 주문서 자동 채움 연결

주문서 페이지에서 아래 정보 사용:

- 기본 수령인 이름
- 기본 연락처
- 기본 주소
- 기본 배송 요청사항

---

## 백엔드 구현 단계

### 1단계. 사용자 프로필 조회/수정 API

필요 API 예시:

- `GET /api/users/me`
- `PATCH /api/users/me`

### 2단계. 배송지 조회/저장 API

초기에는 기본 배송지 1개만 관리해도 된다.

필요 API 예시:

- `GET /api/users/me/addresses`
- `POST /api/users/me/addresses`
- `PATCH /api/users/me/addresses/{addressId}`
- `DELETE /api/users/me/addresses/{addressId}`
- `PATCH /api/users/me/addresses/{addressId}/default`

### 3단계. 주문서와의 연결

주문서 조회 또는 초기 진입 시
기본 배송지를 불러와 프론트 폼에 반영한다.

## 초기 범위와 확장 범위

### 초기 범위

- 헤더 드롭다운
- `/mypage`
- 기본 정보 탭
- 배송지 관리 탭
- 기본 배송지 1개 저장
- 주문서 자동 채움 연결

### 확장 범위

- 배송지 여러 개 저장
- 최근 사용 배송지
- 주문 내역 탭 통합
- 프로필 이미지 업로드
- 비밀번호 변경
- 결제 수단 저장

---

## 화면 설계 포인트

### 기본 정보 탭

느낌:

- 왼쪽에 아바타
- 오른쪽에 정보 폼
- 저장 버튼은 명확하게

### 배송지 관리 탭

느낌:

- "기본 배송지"가 바로 보이게
- 주소가 한눈에 읽히게
- 수정/저장 버튼 분명하게

즉, 복잡한 설정 화면보다
"주문 전에 필요한 정보만 정리하는 생활형 관리 화면"에 가깝게 가는 것이 좋다.

---

## 권장 구현 순서

1. 헤더 사용자 드롭다운
2. 마이페이지 라우트와 기본 레이아웃
3. 기본 정보 탭 UI
4. 배송지 관리 탭 UI
5. 프로필/배송지 API 연결
6. 주문서 페이지와 기본 배송지 자동 채움 연결
7. 이후 주문 내역 연계

---

## 최종 정리

프로필 페이지는 단순 부가 기능이 아니라
주문 UX를 좋게 만들기 위한 선행 단계다.

가장 자연스러운 구조는 아래다.

1. 헤더 사용자명 클릭
2. 드롭다운에서 프로필 진입
3. 마이페이지 좌측 탭
   - 기본 정보
   - 배송지 관리
4. 배송지 저장
5. 주문서에서 자동 채움

즉, 주문서에서 정보를 직접 처음부터 쓰게 하기보다
프로필 페이지에서 미리 관리하게 하는 쪽이 더 교과서적이고 UX도 낫다.

# 권한 계획 프리뷰

이 문서는 권한 관리 기능을 실제 구현하기 전에 화면 흐름과 운영 관점에서 미리 점검하는 프리뷰 문서입니다.

## 한 줄 요약

- 일반 회원은 쇼핑몰 이용만 합니다.
- 관리자는 관리자 페이지 전체를 사용합니다.
- 회원 관리 메뉴에서 회원 권한을 바로 변경할 수 있게 합니다.
- 권한 변경은 단순하게 보이더라도 운영 사고 방지 규칙을 반드시 함께 둡니다.

## 현재 방향이 괜찮은 이유

- 쇼핑몰 규모가 크지 않아 역할을 세분화할 필요가 아직 없습니다.
- 운영 메뉴가 한 화면 안에 모여 있어 `ADMIN` 하나로 묶는 편이 자연스럽습니다.
- 관리자 화면에서 회원 권한을 직접 바꾸게 하면 별도 복잡한 승인 플로우 없이도 운영이 가능합니다.
- 관리자 여부만 빠르게 판단하면 프론트와 백엔드 모두 구현 부담이 줄어듭니다.

## 화면 기준 프리뷰

현재 관리자 레이아웃 기준으로는 왼쪽 사이드 메뉴에 `회원 관리`를 추가하는 방식이 가장 자연스럽습니다.

이 메뉴에서 기대하는 흐름은 아래와 같습니다.

1. 관리자가 `회원 관리` 메뉴로 이동합니다.
2. 회원 목록을 확인합니다.
3. 각 회원의 현재 권한을 확인합니다.
4. 필요한 경우 `USER`를 `ADMIN`으로 올리거나, `ADMIN`을 `USER`로 내립니다.
5. 변경 후 즉시 목록에 반영됩니다.

즉, 별도 복잡한 회원 상세 화면 없이 목록 중심 UI로 먼저 시작해도 충분합니다.

## 추천하는 화면 요소

- 회원 이름
- 아이디 또는 이메일
- 가입일
- 현재 권한 뱃지
- 권한 변경 버튼 또는 셀렉트
- 변경 확인 모달

가능하면 관리자 계정은 색상 또는 라벨로 눈에 띄게 구분하는 것이 좋습니다.

## 꼭 막아야 하는 운영 사고

### 1. 자기 자신 강등

- 현재 로그인한 관리자가 자기 자신을 `USER`로 바꾸지 못하게 해야 합니다.
- 이 문제를 막지 않으면 관리자 페이지 접근이 갑자기 끊길 수 있습니다.

### 2. 마지막 관리자 삭제

- 시스템의 마지막 `ADMIN` 계정을 일반 회원으로 내리지 못하게 해야 합니다.
- 마지막 관리자가 사라지면 운영 복구가 번거로워집니다.

### 3. 프론트만 믿는 권한 처리

- 버튼 숨김만으로는 충분하지 않습니다.
- 권한 변경 API는 백엔드에서 반드시 다시 검증해야 합니다.

### 4. 실수 클릭으로 인한 권한 변경

- 버튼 한 번으로 바로 바꾸기보다 확인 절차를 두는 편이 안전합니다.

## 이 방향에서 구현하면 좋은 점

- 권한 구조가 단순해 빠르게 개발할 수 있습니다.
- 운영자가 회원 상태를 한곳에서 바로 관리할 수 있습니다.
- 관리자 메뉴 전체를 하나의 기준으로 제어할 수 있어 보안 흐름이 명확합니다.
- 나중에 권한을 세분화해야 할 때도 현재 구조를 기반으로 확장할 수 있습니다.

## 아직 과하게 만들 필요 없는 것

현재 단계에서는 아래 항목까지 한 번에 넣지 않아도 됩니다.

- 관리자 역할 세분화
- 권한 변경 이력 화면
- 승인 워크플로우
- 부서별 운영 권한

이 기능들은 실제 운영에서 필요성이 생긴 뒤 추가해도 늦지 않습니다.

## 구현 전 체크 포인트

- 사용자 엔티티에 role 필드가 있는지 확인
- role enum을 `USER`, `ADMIN`으로 단순하게 유지할 수 있는지 확인
- `/admin/**` 보호를 백엔드에서 처리하는지 확인
- 로그인 사용자 role을 프론트에서 사용할 수 있는지 확인
- 회원 관리 메뉴와 회원 목록 화면을 어디에 배치할지 확정

## 결론

현재 프로젝트에서는 `회원 관리` 메뉴를 관리자 사이드에 두고, 그 안에서 회원 권한을 바로 변경하게 하는 방향이 적절합니다.

다만 이 기능은 단순한 UI 문제가 아니라 운영 안전장치가 같이 들어가야 하는 기능입니다. 그래서 화면은 단순하게 가더라도, 자기 자신의 권한 변경 금지와 마지막 관리자 보호 규칙은 처음부터 함께 넣는 것이 좋습니다.

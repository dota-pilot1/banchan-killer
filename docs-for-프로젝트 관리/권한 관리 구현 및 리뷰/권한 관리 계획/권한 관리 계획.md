# 권한 관리 계획

이 문서는 현재 반찬 쇼핑몰 프로젝트에서 적용할 권한 관리 구조를 정리한 계획 문서입니다.

## 목표

- 복잡한 권한 체계 대신 `USER`, `ADMIN` 2단계 역할 구조로 시작합니다.
- 일반 회원은 쇼핑몰 이용에 필요한 기능만 사용합니다.
- 관리자는 관리자 페이지 전체에 접근할 수 있도록 구성합니다.
- 관리자 화면에서 회원 권한을 바로 변경할 수 있게 하되, 운영 사고를 막는 안전장치를 함께 둡니다.

## 기본 역할 구조

### 1. USER

- 상품 조회
- 카테고리 탐색
- 장바구니 사용
- 주문 및 결제
- 본인 주문 조회
- 본인 정보 확인 및 수정

### 2. ADMIN

- 관리자 페이지 전체 접근
- 상품 등록, 수정, 삭제
- 주문 상태 확인 및 관리
- 회원 목록 조회
- 회원 권한 변경
- 배너, 카테고리, 리뷰 등 운영 메뉴 접근

## 현재 단계에서 역할을 단순하게 두는 이유

- 프로젝트 규모가 크지 않아 세분화된 권한이 아직 필요하지 않습니다.
- `상품 관리자`, `주문 관리자`, `회원 관리자`처럼 역할을 미리 나누면 구현 복잡도만 올라갑니다.
- 현재는 `관리자 여부`만 명확히 판단해도 운영 기능을 충분히 제어할 수 있습니다.
- 이후 운영 범위가 커질 때만 역할을 추가하는 방식이 더 안전합니다.

## 관리자 접근 정책

- `/admin/**` 경로는 `ADMIN`만 접근 가능하도록 처리합니다.
- 일반 회원이 관리자 URL로 직접 접근해도 백엔드에서 차단해야 합니다.
- 프론트에서도 관리자 메뉴와 관리자 페이지 진입을 숨기거나 가드 처리하지만, 최종 권한 검사는 반드시 백엔드에서 수행합니다.

## 회원 관리 메뉴 계획

- 관리자 사이드 메뉴에 `회원 관리` 항목을 둡니다.
- `회원 관리` 페이지에서는 회원 목록을 조회할 수 있어야 합니다.
- 각 회원의 현재 권한을 확인하고 `USER`, `ADMIN`을 변경할 수 있어야 합니다.
- 권한 변경은 회원 상세 페이지 없이 목록에서 바로 처리해도 됩니다.

## 회원 관리 화면 구성 예시

- 회원 이름
- 로그인 아이디 또는 이메일
- 가입일
- 현재 권한
- 권한 변경 버튼 또는 셀렉트 박스

관리자 계정은 뱃지나 라벨로 구분해 표시하면 운영자가 상태를 빠르게 확인할 수 있습니다.

## 권한 변경 시 필수 안전장치

### 1. 자기 자신의 권한 변경 금지

- 현재 로그인한 관리자는 자기 자신의 권한을 `USER`로 변경할 수 없게 합니다.
- 이 제약이 없으면 관리자 화면 접근 권한을 스스로 잃는 사고가 발생할 수 있습니다.

### 2. 마지막 관리자 보호

- 시스템에 `ADMIN` 계정이 1명만 남아 있는 경우, 그 계정을 `USER`로 변경하지 못하게 막아야 합니다.
- 관리자 계정이 0명이 되면 운영 기능 전체가 막히는 문제가 발생합니다.

### 3. 권한 변경 확인 절차

- 권한 변경 시 즉시 반영하기보다 확인 모달 또는 확인 버튼을 한 번 더 거치게 합니다.
- 실수로 잘못 클릭해 권한이 바뀌는 상황을 줄일 수 있습니다.

### 4. 백엔드 검증 우선

- 프론트에서 버튼을 숨기는 것만으로는 충분하지 않습니다.
- 권한 변경 API 내부에서 현재 요청 사용자의 권한, 대상 회원, 변경 가능 여부를 모두 다시 검증해야 합니다.

## 백엔드 구현 방향

- 사용자 엔티티에 `role` 필드를 둡니다.
- `role`은 enum으로 관리합니다.
- enum 값은 우선 `USER`, `ADMIN`만 둡니다.
- Spring Security 인증 정보에서도 이 role 값을 기반으로 권한을 부여합니다.

예시 방향:

1. 사용자 조회 시 role 포함
2. 로그인 성공 시 role을 인증 정보에 반영
3. `/admin/**` 요청은 `ADMIN`만 허용
4. 회원 권한 변경 API는 `ADMIN`만 호출 가능

## 프론트 구현 방향

- 로그인 사용자 정보에 role 값을 포함합니다.
- role이 `ADMIN`일 때만 관리자 진입 버튼 또는 관리자 메뉴를 노출합니다.
- 관리자 화면의 사이드 메뉴에 `회원 관리`를 추가합니다.
- 회원 목록 화면에서 권한 변경 UI를 제공합니다.
- 변경 후에는 목록을 다시 조회하거나 로컬 상태를 갱신해 즉시 반영합니다.

## 나중에 확장할 수 있는 구조

현재는 단순하게 시작하되, 구조는 확장 가능하게 두는 것이 좋습니다.

향후 필요 시 아래처럼 역할을 늘릴 수 있습니다.

- `PRODUCT_ADMIN`
- `ORDER_ADMIN`
- `CS_ADMIN`
- `SUPER_ADMIN`

다만 지금 단계에서는 위 역할들을 실제로 만들지 않고, enum과 접근 제어 구조만 확장 가능하게 두는 수준이면 충분합니다.

## 결론

- 현재 프로젝트는 `USER`, `ADMIN` 2단계 역할 구조로 시작하는 것이 적절합니다.
- 관리자 페이지 전체는 `ADMIN`만 접근 가능하게 두면 됩니다.
- 사이드 메뉴의 `회원 관리`에서 회원 권한을 바로 변경할 수 있게 해도 됩니다.
- 대신 자기 자신 권한 변경 금지, 마지막 관리자 보호, 백엔드 검증은 반드시 포함해야 합니다.

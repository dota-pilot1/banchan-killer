# 주문 및 결제 기능 구현 계획

## 1. 현재 상태 요약

- 현재 DB 기준 핵심 테이블은 `products`, `users` 중심이다.
- 상품은 이미 재고(`stock_quantity`), 상태(`SALE`, `SOLDOUT`, `HIDDEN`)를 가지고 있다.
- 프론트는 상품 조회와 관리자 상품 수정 화면까지 구현되어 있다.
- 아직 주문, 장바구니, 배송지, 결제, 주문 상태 변경 이력 도메인은 없다.

즉, 지금 단계에서는 단순히 `orders` 테이블 하나만 추가하는 방식보다,
주문 생성부터 결제 승인, 재고 차감, 관리자 주문 처리까지 이어지는 흐름을 먼저 정하고
그 흐름에 맞춰 DB와 API를 설계하는 게 맞다.

---

## 2. 먼저 정해야 할 운영 기준

구현 전에 아래 기준을 먼저 고정하는 것이 좋다.

### 2-1. 주문 상태 흐름

권장 흐름:

- `PENDING_PAYMENT`: 주문서 생성, 아직 결제 전
- `PAID`: 결제 승인 완료
- `PREPARING`: 상품 준비 중
- `SHIPPED`: 배송 시작
- `DELIVERED`: 배송 완료
- `CANCELLED`: 주문 취소
- `REFUNDED`: 환불 완료

### 2-2. 결제 상태 흐름

권장 흐름:

- `READY`: 결제 시도 전
- `REQUESTED`: 결제창 호출 후 승인 대기
- `DONE`: 결제 승인 완료
- `FAILED`: 결제 실패
- `CANCELLED`: 결제 취소
- `PARTIAL_CANCELLED`: 부분 취소가 있으면 확장

### 2-3. 재고 차감 시점

초기 버전에서는 아래처럼 단순하게 가는 것이 안전하다.

- 주문 생성 시 재고 차감하지 않음
- 결제 승인 완료 시 재고 차감
- 결제 실패/취소 시 재고 유지
- 결제 완료 후 취소/환불 시 재고 복원 여부는 운영 정책으로 결정

이 방식이 초기 구현에서는 가장 단순하고, 결제 미완료 주문 때문에 재고가 오래 묶이는 문제를 줄일 수 있다.

### 2-4. 배송비 정책

초기부터 고정해야 프론트/백엔드 계산이 안 어긋난다.

예시:

- 기본 배송비 `3,000원`
- `40,000원 이상 무료배송`

### 2-5. 주문 가격 계산 기준

주문 테이블에는 당시 가격을 snapshot으로 저장해야 한다.

- 상품 현재 가격을 그대로 참조만 하면 안 됨
- 주문 아이템에 `상품명`, `주문 당시 단가`, `수량`, `소계`를 저장해야 함

## 3. 권장 구현 순서

### 1단계. 주문 도메인 설계

가장 먼저 해야 할 일은 주문 관련 엔티티를 분리해서 만드는 것이다.

필수 후보:

- `carts`
- `cart_items`
- `orders`
- `order_items`
- `payments`
- `delivery_addresses`

초기 버전에서는 `delivery_addresses`를 주문에 직접 포함해도 된다.
다만 재사용 주소록 기능까지 갈 계획이면 별도 테이블이 낫다.

### 2단계. 장바구니 구현

주문보다 먼저 장바구니를 구현하는 것이 보통 자연스럽다.

필수 기능:

- 장바구니 담기
- 수량 변경
- 장바구니 목록 조회
- 장바구니 상품 삭제
- 품절/숨김 상품 예외 처리

이 단계에서 같이 처리할 것:

- `SALE` 상태 상품만 담기 가능
- 장바구니 조회 시 재고 부족 여부 표시
- 비회원 장바구니를 할지, 회원 전용으로 시작할지 결정

현재 구조에서는 회원 기능이 이미 있으므로, 초기 버전은 `회원 장바구니`부터 가는 것이 구현 부담이 낮다.

### 3단계. 주문 생성 API 구현

장바구니에서 선택한 항목으로 주문서를 만드는 단계다.

주문 생성 시 해야 할 일:

- 사용자 정보 확인
- 상품 상태 확인
- 각 상품 재고 확인
- 주문 아이템 생성
- 상품명/가격 snapshot 저장
- 배송비 계산
- 총 금액 계산
- 주문 상태를 `PENDING_PAYMENT`로 생성

주의:

- 이 시점에 재고를 바로 차감할지, 결제 완료 후 차감할지 정책을 고정해야 한다.
- 초기 버전은 결제 승인 후 차감 권장

### 4단계. 결제 연동 전 주문 조회/관리 API 구현

결제 붙이기 전에 주문 자체가 잘 동작해야 한다.

필수 API:

- 내 주문 목록 조회
- 내 주문 상세 조회
- 관리자 주문 목록 조회
- 관리자 주문 상세 조회
- 관리자 주문 상태 변경
- 주문 취소 요청

이 단계까지 되면 결제를 빼더라도 주문 흐름 검증이 가능하다.

### 5단계. 결제사 연동

초기 국내 서비스면 보통 `토스페이먼츠`, `아임포트(포트원)` 같은 PG 연동을 고려하게 된다.

권장 순서:

1. 프론트에서 결제창 호출
2. 결제 완료 후 PG가 전달한 `paymentKey`, `orderId`, `amount` 수신
3. 백엔드에서 PG 승인 API 호출
4. 승인 성공 시 `payments` 저장
5. 같은 트랜잭션 또는 보장된 후속 처리에서 주문 상태 `PAID` 변경
6. 재고 차감

중요 포인트:

- 결제 승인 검증은 프론트만 믿으면 안 되고, 백엔드에서 PG와 직접 확인해야 한다.
- `orderId`는 외부 노출용 주문번호로 별도 생성하는 것이 좋다.
- 결제 승인 요청의 `amount`는 서버가 계산한 주문 금액과 반드시 비교해야 한다.

### 6단계. 관리자 주문 처리 화면 구현

운영 단계에서 꼭 필요하다.

관리자 기능 예시:

- 주문 목록 필터링
- 상태별 주문 확인
- 주문 상세에서 상품/수량/금액/배송지 확인
- `PAID -> PREPARING -> SHIPPED -> DELIVERED` 변경
- 취소/환불 처리

현재 관리자 좌측 메뉴에 `주문 관리`가 예정 상태로 보이므로,
이 화면은 상품 관리 다음 우선순위로 자연스럽다.

### 7단계. 취소/환불 정책 반영

초기 버전에서는 전체 취소만 먼저 구현하는 편이 안전하다.

권장 범위:

- 결제 완료 전 취소
- 결제 완료 후 배송 전 취소
- 관리자 환불 승인
- 재고 복원 처리

부분 취소, 부분 환불은 복잡도가 크게 올라가므로 2차 확장으로 두는 것이 좋다.

## 8. 지금 코드 기준으로 먼저 손봐야 할 점

현재 프로젝트에는 아래 선행 정리가 필요하다.

### 8-1. 상품 카테고리 기준 정리

AGENTS 규칙 기준 메인 카테고리는 아래 3개다.

- `야채 반찬`
- `고기 반찬`
- `김치류`

현재 백엔드 enum은 더 넓게 잡혀 있다.

- `SIDE_DISH`
- `SOUP_STEW`
- `MAIN_COURSE`
- `KIMCHI`
- `SEASONAL`

이 자체는 문제는 아니지만,
주문/장바구니/관리자 화면에서 카테고리 표기를 쓸 일이 늘어나므로
프론트 라우트(`/category/vegetable`, `/category/meat`, `/category/kimchi`)와
백엔드 매핑 규칙을 더 명확히 유지해야 한다.

### 8-2. 주문번호 분리

내부 PK `id`와 별도로 외부 노출용 `orderNumber`를 두는 것이 맞다.

이유:

- 결제사 연동 시 사용하기 좋음
- 고객 문의 응대가 쉬움
- 순차 PK 직접 노출을 피할 수 있음

### 8-3. BaseEntity 감사 컬럼 점검

현재 `createdAt`, `updatedAt`는 `AuditingEntityListener`를 사용하고 있다.
주문/결제 엔티티에도 그대로 활용 가능하지만,
실제 auditing 활성화 설정이 되어 있는지 한 번 점검하는 것이 좋다.

## 9. 현실적인 개발 단계 제안

- 결제 실패 재시도
- 장바구니 만료/예약재고

## 10. 결론

대략적인 절차는 아래 순서가 가장 무난하다.

1. 주문 상태/결제 상태/재고 차감 시점 정책 확정
2. `cart`, `order`, `order_item`, `payment` DB 설계
3. 장바구니 API/화면 구현
4. 주문 생성 API/주문 조회 API 구현
5. 관리자 주문 관리 API/화면 구현
6. PG 결제 승인 연동
7. 취소/환불과 재고 복원 처리 추가
8. 테스트와 예외 케이스 보강

즉, `주문 -> 결제`가 아니라,
실제로는 `장바구니 -> 주문 생성 -> 결제 승인 -> 재고 차감 -> 관리자 처리` 순서로 구현하는 것이 맞다.

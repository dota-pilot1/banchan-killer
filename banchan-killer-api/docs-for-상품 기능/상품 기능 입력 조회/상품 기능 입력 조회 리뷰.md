# 상품 기능 구현 코드 리뷰 (DDD & Clean Code)

본 문서는 상품 등록 및 조회 기능의 아키텍처와 코드 품질을 DDD 및 클린 코드 원칙에 따라 리뷰한 결과입니다.

## 1. DDD (Domain-Driven Design) 준수 사항

### ✅ 1-1. 계층형 아키텍처 (Layered Architecture)의 명확한 분리
*   **Presentation (`ProductController`)**: HTTP 요청/응답 처리 및 외부 노출 API 정의에만 집중합니다.
*   **Application (`ProductService`)**: 트랜잭션 관리와 흐름 제어를 담당하며, 구체적인 비즈니스 로직은 도메인에 위임할 준비가 되어 있습니다.
*   **Domain (`Product`, `ProductRepository`)**: 외부 기술(JPA 등)에 의존하지 않는 순수 도메인 로직의 핵심입니다.
*   **Infrastructure (`ProductRepositoryImpl`, `ProductJpaRepository`)**: DB와 같은 외부 저장소와의 구체적인 통신을 담당합니다.

### ✅ 1-2. DIP (의존성 역전 원칙) 적용
*   도메인 계층에 `ProductRepository` 인터페이스를 두고, 인프라 계층에서 이를 구현하게 함으로써 **도메인이 데이터베이스 기술에 종속되지 않도록** 설계되었습니다. 이는 추후 DB를 교체하거나 테스트 시 Mock을 사용하기 매우 유리한 구조입니다.

### ✅ 1-3. 도메인 엔티티 보호
*   `Product` 엔티티의 생성자를 `PROTECTED`로 제한하고 `@Builder`를 사용하여, 무분별한 객체 생성을 막고 **의미 있는 생성 방식**을 강제했습니다.

---

## 2. 클린 코드 (Clean Code) 및 설계 품질

### ✅ 2-1. Java Record 활용 (DTO)
*   `ProductRequest`와 `ProductResponse`에 Java 16+의 **Record**를 사용하여 불변성(Immutability)을 확보하고 코드를 비약적으로 간소화했습니다.

### ✅ 2-2. Bean Validation (JSR-303)
*   DTO 계층에서 `@NotBlank`, `@Min`, `@NotNull` 등을 사용하여 유효하지 않은 데이터가 서비스 레이어까지 침범하지 못하도록 **fail-fast**하게 설계되었습니다.

### ✅ 2-3. 의미 있는 예외 처리
*   조회 실패 시 `IllegalArgumentException`을 던짐으로써 에러의 원인을 명확히 하고, 이를 글로벌 예외 핸들러에서 처리할 수 있는 구조를 갖췄습니다.

---

## 3. 개선 제언 (고도화 포인트)

*   **Custom Exception**: 현재 `IllegalArgumentException`을 사용 중이나, `ProductNotFoundException` 같은 전용 예외를 정의하면 도메인 언어(Ubiquitous Language)를 더 잘 표현할 수 있습니다.
*   **Category Enum**: 현재 카테고리가 `String`으로 되어 있는데, 고정된 카테고리가 있다면 이를 `Enum`으로 관리하여 오타로 인한 런타임 에러를 방지하는 것이 좋습니다.
*   **Pagination**: 상품 목록이 많아질 경우를 대비해 `Pageable`을 사용한 페이징 조회를 도입할 준비가 필요합니다.
